SEON Docs Evaluation Report

An LLM built a complete fraud-screened checkout from scratch using only the SEON documentation. This report documents exactly what happened, what it cost in tokens, and where the docs helped or hurt.

7/10

Good documentation — not LLM-optimised, but a viable integration path

SEON's docs contain everything an LLM needs to build a working integration, but that information is spread across 5+ separate pages with no machine-readable entry point, no structured flow diagram, and critical steps buried or implied. An LLM can succeed here, but it requires roughly 5× more tokens than a purpose-built LLM-friendly docs site would need.

⚠ Not LLM-optimised — no llms.txt, no OpenAPI spec linked, no single-page flow reference

🚀 How to use this demo

  1. Go to the Store and pick any product.

  2. Fill the checkout form. Use any email and a real 6-digit card BIN — try 424242 (Visa), 555555 (Mastercard), 378282 (Amex). Phone is optional but improves accuracy.

  3. Hit Place order & pay. The SEON JS Agent captures your device fingerprint + behavioural signals, then a Vercel serverless function POSTs to the SEON Fraud API with your key kept server-side.

  4. A result modal shows APPROVE, REVIEW, or DECLINE with the fraud score, email domain, IP country, VPN status, and card type.

  5. Check the SEON Admin sandbox — the transaction appears in real time, including device fingerprint and rule details.

🔧 What was built full integration scope

Starting from an empty folder, the LLM read SEON's documentation, designed the architecture, and built the entire project — store, checkout, serverless proxy, and this report — in a single session. No prior knowledge of SEON was used; everything was derived from the docs.

Browser (checkout.html)
  ├─ SEON JS Agent v6 (cdn.seondf.com)
  │   ├─ seon.init() ← on page load, behavioural tracking starts
  │   └─ seon.getSession() ← on submit, returns encrypted base64 session token
  │
  ├─ IP detection (api.ipify.org)
  │
  └─ POST /api/seon-check { email, ip, card_bin, session, amount, sku }

Vercel Serverless Function (api/seon-check.js)
  ├─ Reads SEON_API_KEY from env (never exposed to browser)
  └─ POST https://api.seon.io/SeonRestService/fraud-api/v2/
      Headers: X-API-KEY, Content-Type: application/json
      action_type: "purchase"

SEON Response
  ├─ data.state → APPROVE / REVIEW / DECLINE
  ├─ data.fraud_score (0–100)
  ├─ data.email_details { email_domain, deliverable, … }
  ├─ data.ip_details { ip_country, ip_type, vpn, … }
  └─ data.card_details { card_country, card_type, … }

1

Orientation — what exists?

Read the docs home page and the API reference index to build a mental map of the product. This was fast and clear — the structure is well-organised.

Core API — first crash: page too large

Attempted to read the Fraud API page as plain text. The page returned 55,409 characters — exceeding the tool limit. Had to fall back to reading the accessibility tree at reduced depth, losing code example content in the process. This cost a second request and partial data.

/api-reference/fraud-api (FAILED) /api-reference/fraud-api (read_page fallback)
3

Use-case guidance — cross-referencing two pages

To know which fields to send for an ecommerce purchase, the LLM had to read both the Integration Guide and the Introduction separately. Neither page alone had the complete answer — the Integration Guide listed field categories, the Introduction had the score thresholds and 3-step model. The LLM had to synthesise across both.

/integration/api-integration-guide /api-reference/introduction
4

JS Agent — found late, by accident

The SEON JS Agent (device fingerprinting) is a prerequisite for the session field in the Fraud API payload. The LLM discovered it only because the Fraud API page happened to include it as a deep subsection. If the LLM had only read the Quick Start payload example, the session field would have been silently omitted — and the integration would have worked but with no device data, with no error to diagnose it.

Not in Quick Start Not in Integration Guide Buried in /api-reference/fraud-api
5

CORS / browser calls — undiscovered until build time

The docs contain no mention of CORS policy. When the LLM designed a browser-direct integration (as requested), it had to assume server-side proxying was needed — and built a Vercel serverless function as a precaution. This was the right call, but it was based on inference, not documentation. An integrator without that background knowledge would have been blocked by a confusing network error with no docs to reference.

No CORS documentation anywhere
6

action_type — values never formally listed

The LLM needed to know the correct value for an ecommerce purchase. The word purchase appeared in prose examples, and error code 3007 says "Invalid action_type — remove spaces and special characters" but still doesn't list the valid values. The LLM used "purchase" based on the prose example — but this is inference, not a spec.

No action_type enum table Inferred from prose in /api-reference/introduction
7

Integration Process — misleading page title

Searching for developer onboarding steps, the LLM navigated to /integration/integration-process. This page describes a 7-step enterprise sales onboarding (discovery call → business discussion → proposal → PoV → kick-off), not a developer integration checklist. A developer following this path would spend 300 tokens learning about "Solution Alignment" meetings before realising the page isn't relevant to their task.

/integration/integration-process (sales page, not dev guide)
8

No llms.txt — checked and confirmed absent

The LLM checked https://docs.seon.io/llms.txt — a convention used by LLM-optimised sites to provide a structured, token-efficient summary of the entire docs. The file does not exist. Instead the LLM received a generic sitemap page. This is the single highest-leverage improvement SEON could make for LLM-assisted integrations.

✅ What works well

Strength Clear 3-step mental model
The Introduction page's Send data → Enrich & Score → Feedback structure is immediately graspable. An LLM can extract the workflow in a single read and generate structurally correct integration code without guessing the flow.
Source /api-reference/introduction — "At its core, our fraud prevention platform operates in three simple steps: You send user / transaction / device data. We enrich the data and deliver a risk score based on rules. You give feedback on the results."
Strength Score thresholds are unambiguously tabulated
APPROVE (0–10) / REVIEW (10–20) / DECLINE (20+) in a table with no prose interpretation needed. This is exactly the kind of unambiguous numeric data that eliminates hallucination risk in generated decision logic.
Source /api-reference/introduction — "State: APPROVE, Threshold: 0–10 / REVIEW: 10–20 / DECLINE: 20+"
Strength Industry + use-case Quick Start selector
Selecting Ecommerce + Payment + Python instantly filtered the payload to the relevant fields. For a human developer this is great UX. For an LLM, however, this selector is rendered by JavaScript — meaning a standard web fetch won't see the filtered content. The LLM had to use a full browser session to access it.
Caveat JS-rendered widgets are invisible to basic HTTP fetchers. An llms.txt version of this filtered content would be significantly more useful for AI-assisted integrations.
Strength Authentication is simple and standard
X-API-KEY: [license_key] is a well-understood pattern. No OAuth dance, no token refresh, no scope negotiation. An LLM generates correct auth code from a single sentence. The sandbox/production key separation is also clearly explained.
Strength Modular API with opt-in enrichment
The config object (ip_api, email_api, phone_api, etc.) lets an LLM generate a minimal working payload without accidentally enabling slow or expensive enrichments. The "never fail the API call" philosophy (missing enrichment returns 200 with empty fields) is well-explained and makes error handling straightforward.
Source Fraud API page — "Our general rule is to never fail the Fraud API call. We return 200 OK, and just not populate the fields."
Strength Comprehensive Label API with typed fraud categories
The Label API has an extensive, well-categorised set of fraud labels (ecommerce_chargeback_fraud, fraud_detection_stolen_card, fraud_detection_bot, etc.) across vertical-specific groups. This level of ML feedback granularity is impressive and the values are clearly enumerated — unlike action_type.
Strength Webhook system is thorough and well-documented
HMAC-SHA256 signature verification, retry table with exact delays, 20+ event types, idempotency via Request-Id, and legacy migration warnings. This is production-ready documentation. An LLM can build a complete webhook consumer from this page alone.
Strength Regional endpoint documentation
EU, US (East), Middle East, and APAC endpoints are documented. This is important for compliance and latency — many APIs bury this or omit it entirely.
Source Fraud API page — EU: api.seon.io, US: api.us-east-1-main.seon.io, ME: api.me-central-1-main.seon.io, APAC: api.ap-southeast-1-main.seon.io

⚠️ What can be improved found during the actual build

Critical No llms.txt or machine-readable docs entry point
The llms.txt convention provides a single, token-efficient structured summary of an entire docs site — purpose-built for LLMs. SEON has no such file. An LLM integrating SEON must make 8–11 separate HTTP requests to gather equivalent information, consuming roughly 7× more tokens than necessary. This also makes SEON invisible to AI coding assistants that check for llms.txt before attempting an integration.
Confirmed GET https://docs.seon.io/llms.txt returned a generic sitemap page, not a structured LLM resource file.
Fix: Add /llms.txt with a structured summary: auth method, core endpoints + HTTP methods, field names, action_type enum, score thresholds, JS Agent CDN URL, and regional endpoints. Effort: ~2 hours. Impact: reduces LLM token cost by ~80%.
High impact No end-to-end flow page — must synthesise across 5 pages
To understand a complete purchase transaction flow, the LLM had to read: Introduction (3-step model) → API Integration Guide (which fields for ecommerce) → Fraud API (request/response) → JS Agent section (device fingerprinting) → Label API (feedback). There is no single canonical page, and no sequence diagram showing the full request lifecycle.
Impact observed: Caused the JS Agent prerequisite to nearly be missed. The Fraud API page shows a session field in the payload with no inline explanation — only discoverable by reading 80% down the same page.
Fix: Add a "Complete integration walkthrough" page with a sequence diagram: Load JS Agent → seon.init() → user action → seon.getSession() → POST /fraud-api/v2/ → read state → POST /label-api. One page, one flow. Covers 80% of new integration needs.
High impact JS Agent setup is buried, underpromoted, and fails silently
The JS Agent is a frontend prerequisite for the session field in every Fraud API call — yet it appears only as a subsection of the Fraud API reference page, is absent from the Quick Start, and is not mentioned in the Integration Guide's "Suggested setup" checklists. More critically: if you send the Fraud API request without a session token, you get a perfectly valid 200 response — just with no device data. There is no error, no warning, no indication the integration is incomplete.
Source Fraud API page — "The Fraud API call should still be executed if the session is missing." This is good resilience design but terrible discovery design — the silent fallback masks the missing step.
Fix: Add a callout box at the top of the Fraud API page: "Using device fingerprinting? You must load the JS Agent before calling this API." Promote JS Agent to a top-level nav item. Add to Quick Start under each web payload example.
High impact Fraud API page is too large — breaks standard LLM fetch tools
The Fraud API page contains 55,409 characters (~13,800 tokens), exceeding standard context extraction limits. The LLM's first attempt to read it failed outright and required a fallback approach. A page of this size covering request fields, response fields, JS Agent, iOS SDK, Android SDK, eKYC examples, and AML examples on one page is too dense for a single-page extraction.
Error observed: "Output exceeds 50000 character limit (55,409 characters). Try using read_page with a specific ref_id."
Fix: Split the Fraud API page into focused sub-pages: /fraud-api/request-reference, /fraud-api/response-reference, /fraud-api/js-agent, /fraud-api/mobile-sdks, /fraud-api/ekyc, /fraud-api/aml-payload. Each can be fetched independently and stays under 15k chars.
Medium impact action_type valid values are never enumerated
One of the most fundamental fields in every Fraud API request — action_type — has no reference table of valid values anywhere in the docs. They appear only in prose examples (account_register, account_login, purchase, deposit, withdrawal). Error code 3007 says "Invalid action_type — remove spaces and special characters" but still doesn't list what valid values are.
Error code 3007: "Invalid action_type — Remove spaces and special characters" — but no list of valid values provided.
Fix: Add a table to the Fraud API request section: action_type | When to use | Notes. Five rows. Could also appear in the Quick Start. This single table eliminates a common integration mistake.
Medium impact No full realistic JSON response example
The response documentation describes fields but contains no complete JSON response for a typical APPROVE transaction. An LLM building response-parsing code must guess field paths from the field descriptions. The actual path is response.data.state, response.data.email_details.deliverable etc. — a common hallucination point is response.state (wrong) vs response.data.state (correct).
Fix: Add collapsible full JSON examples for APPROVE / REVIEW / DECLINE with realistic-looking values. This is standard in high-quality API docs (Stripe, Twilio, Plaid). A realistic example is more useful to an LLM than 50 field descriptions.
Medium impact "Integration Process" page is a sales onboarding guide, not a dev guide
The page at /integration/integration-process describes a 7-step enterprise sales process (Discovery Call → Business Discussion → Solution Alignment → Proposal → PoV → Kick-Off → Go-Live). A developer searching for "how to get started with the integration" would land on this page and spend tokens learning about account executives and blackout periods before realising it's not relevant to their task.
Step 1 on the Integration Process page: "Discovery Call – Account Executive assesses your primary needs and challenges…"
Fix: Rename the page to "Onboarding & Sales Process" or move it out of the Integration section into a "Getting Started" or "Enterprise" section. Add a separate "Developer Quickstart" as the first item in the Integration nav.
Medium impact CORS / browser-vs-server architecture undocumented
The docs contain no mention of CORS policy. All examples use server-side code but no warning is given against browser-direct calls. The LLM was explicitly asked to build a browser-only integration and had to fall back on inferred best practice (server-side proxy). An integrator without that background knowledge would hit a confusing network error with no documentation to help diagnose it.
Fix: One sentence in the Introduction: "The SEON API is designed for server-side use. Always proxy requests through your backend to protect your API key and avoid CORS restrictions." Zero effort, prevents significant confusion.
Low impact Quick Start selector is JavaScript-rendered — invisible to LLM fetch tools
The industry + use-case Quick Start is one of SEON's best features, but it's rendered by JavaScript. A basic HTTP fetch of the Introduction page returns the selector UI — not the filtered content. An LLM using standard web fetch tools gets the scaffolding, not the payload. Only a full browser session can access the actual content. This undermines the feature's value for AI-assisted integrations.
Fix: Add a static fallback page at /quick-start/ecommerce-purchase (and other combinations) with the pre-filtered payload as plain HTML. Or include pre-rendered versions in llms.txt.
Low impact Session token lifecycle not specified
The JS Agent docs don't specify whether the session token is single-use, time-limited, or reusable. For a multi-step checkout (e.g., show summary → confirm → pay), a developer doesn't know whether to call getSession() once or on each step. The LLM called it once on submit — inferred, not documented.
Fix: One sentence: "Each getSession() call returns a unique, single-use token. Call it once per transaction, immediately before submitting."
Low impact Rate limits and error codes not cross-linked from Fraud API page
Rate limits (2 req/s trial, 10 req/s paid, 429 on breach) are documented only in the Introduction. Error codes are on a separate page. Neither is linked from the Fraud API page. An LLM reading only the Fraud API reference has no indication it needs to handle 429 responses.
Fix: Add a small "Errors & Rate Limits" callout box at the bottom of the Fraud API page with the key codes (2001 auth, 3007 invalid field, 4004 rate limit) and a link to the full errors page.

🎯 Prioritised recommendations

  1. Add /llms.txt

    Structured LLM entry point: auth method, core endpoint URLs, action_type enum, score thresholds, JS Agent CDN URL, regional endpoints. Reduces LLM token cost from ~27k to ~4k. Makes SEON visible to AI coding assistants. Biggest ROI of any docs change.

    Effort: Low (~2–4 hours)

  2. Add a single end-to-end integration walkthrough page

    One page covering the full lifecycle with a sequence diagram: JS Agent load → seon.init() → event → seon.getSession() → POST /fraud-api → read state → POST /label-api. Eliminates the need to synthesise across 5 pages.

    Effort: Medium (~1–2 days)

  3. Promote JS Agent to a first-class integration step

    Add it as Step 1 on the walkthrough page. Add a callout box at the top of the Fraud API page. Add to the Quick Start. The silent failure mode makes this uniquely dangerous to omit.

    Effort: Low (~half day)

  4. Split the Fraud API page (55k chars → multiple sub-pages)

    Request reference, response reference, JS Agent, mobile SDKs, eKYC, AML as separate pages. Each stays under 15k chars and can be fetched independently by an LLM. Eliminates the page-too-large crash.

    Effort: Medium (~1 day)

  5. Add action_type enum table + full JSON response examples

    5-row table of valid action_type values. Collapsible APPROVE / REVIEW / DECLINE response JSON examples. Two content additions that eliminate major hallucination risks for LLMs generating integration code.

    Effort: Low (~2–3 hours)

  6. Add one-sentence CORS policy statement

    "The SEON API is designed for server-side use. Always proxy requests through your backend." One sentence in the Introduction prevents significant confusion for browser-first developers.

    Effort: Trivial (5 minutes)

  7. Rename / move the "Integration Process" page

    It describes an enterprise sales onboarding flow, not a developer integration. Rename it "Enterprise Onboarding" and move it out of the Integration nav section. Add a "Developer Quickstart" as the first Integration nav item instead.

    Effort: Low (navigation restructure)

🏁 Overall verdict

7/10

Viable — not LLM-optimised

SEON's docs contain everything needed to build a working integration. The mental model is sound, auth is simple, and several features (score thresholds, Label API labels, webhook docs) are genuinely excellent. The friction comes from information density and distribution — not missing information. With llms.txt, a walkthrough page, and a few enumeration tables, SEON could be among the most LLM-friendly fraud APIs available. The gap isn't quality — it's structure.
