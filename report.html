SEON Docs Evaluation Report

An LLM-driven integration assessment: building a real checkout with SEON's Fraud API while documenting every friction point.

Evaluated by Claude Sonnet 4.6 Date: 2026-05-06 Docs: docs.seon.io Stack: Pure HTML/JS + Vercel Serverless

🚀 How to use this demo

  1. Go to the Store and pick any product.

  2. Fill in the checkout form. Use a real email address and enter a 6-digit card BIN — try 424242 (Visa), 555555 (Mastercard), or 378282 (Amex). Phone is optional.

  3. Hit Place order & pay. SEON's JS Agent captures your device fingerprint, then our serverless proxy POSTs to the SEON Fraud API.

  4. A result modal shows the decision: APPROVE, REVIEW, or DECLINE — plus the fraud score and enriched data (email domain, IP country, card type, VPN status).

  5. Check the SEON Admin sandbox dashboard to see the transaction appear in real time.

🔧 What was built

Starting from an empty folder with nothing but the SEON docs website, an LLM (Claude Sonnet 4.6) read the documentation, designed and built a complete mock ecommerce store, and produced this evaluation — all in a single session.


The integration covers the full recommended flow:

  • SEON JS Agent v6 — loaded from CDN, initialized on page load for behavioral biometrics, getSession() called on form submit to capture an encrypted device fingerprint token
  • Client IP detection — fetched from api.ipify.org before the API call
  • Fraud API — POST /SeonRestService/fraud-api/v2/ with action_type: purchase, email, IP, card BIN, session token, purchase amount, and custom fields
  • Vercel serverless proxy (/api/seon-check) — keeps the API key server-side, eliminates CORS, mirrors production patterns
  • Decision rendering — APPROVE / REVIEW / DECLINE result modal with score and key enrichment details

✅ What works well

Strength Clear 3-step mental model
The Introduction page's "Send data → Enrich & Score → Feedback" structure gives an LLM an immediately graspable workflow. This is the single most important structural choice in the docs.
Strength Risk score thresholds are explicitly tabulated
APPROVE (0–10) / REVIEW (10–20) / DECLINE (20+) in a clean table. LLMs generate correct conditional logic from tables — no hallucination risk here.
Strength Industry + use-case Quick Start selector
Choosing "Ecommerce + Payment" instantly filtered the payload to the relevant fields. This is a best-in-class feature — most fraud APIs force you to read a 200-field reference and guess.
Strength Multi-language code examples + Postman collection
cURL, Java, Python, and PHP examples on the Fraud API page give concrete, parseable templates. The Postman collection link lets developers test immediately without crafting requests from scratch.
Strength Modular API — all fields optional, config-driven
The explanation that each sub-API (Email, Phone, IP, BIN) is opt-in via config.api_fields lets an LLM generate a minimal working payload without accidentally enabling slow or expensive enrichments.

⚠️ What can be improved

High impact No single end-to-end integration walkthrough
To understand a complete purchase flow an integrator must synthesize: Introduction → API Integration Guide → Fraud API Reference → JS Agent section → Label API. Each page adds context missing from the others. There is no sequence diagram.
Recommendation: Add a "Complete integration walkthrough" page with a sequence diagram covering JS Agent load → getSession → POST Fraud API → read state → POST Label API. This single change reduces LLM hallucination risk most.
High impact JS Agent is buried and easy to miss
The JS Agent setup is a deep subsection of the Fraud API page — not a top-level nav item, not mentioned in the Quick Start, and not cross-linked from the Integration Guide. The payload shows a session field with no immediate hint where it comes from. Worse, sending without it returns a valid response — the failure is silent.
Recommendation: Add a callout at the top of the Fraud API page: "Using device fingerprinting? You must integrate the JS Agent first. [→ JS Agent setup]". Promote JS Agent to a first-class nav item. Note in the Quick Start that session comes from the JS Agent.
Medium impact No realistic full response JSON example
The response documentation describes fields but includes no complete JSON example for a typical APPROVE transaction. An LLM building response-parsing code must guess field paths — e.g., is it response.state or response.data.state?
Recommendation: Add collapsible full JSON examples for each state (APPROVE, REVIEW, DECLINE) with realistic-ish data. Standard practice in high-quality API docs (Stripe, Twilio).
Medium impact CORS / browser-vs-server architecture undocumented
The docs don't mention whether the SEON API supports direct browser calls. All examples are server-side but no warning is given. Attempting a browser-direct integration (like this demo) may silently fail or produce confusing network errors.
Recommendation: Add one sentence in the Introduction: "The SEON API is designed for server-side use. Direct browser calls may be blocked by CORS. Always proxy requests through your backend."
Medium impact action_type values not enumerated in one place
Valid values (account_register, account_login, purchase, deposit, withdrawal) are scattered in prose. An LLM may hallucinate plausible but wrong values (e.g., payment instead of purchase).
Recommendation: Add a small reference table of all action_type values with a one-line "when to use" description in the Fraud API request section.
Low impact Feedback loop (Label API) framed as optional
Step 3 reads as best-practice guidance. New integrators routinely skip it, which degrades ML model quality over time.
Recommendation: Frame labeling as part of the core integration checklist with a "mandatory vs optional" distinction.
Low impact Session token lifecycle not specified
The JS Agent docs don't say whether the session token is single-use or reusable. A multi-step checkout flow developer doesn't know if they need to call getSession() once or per request.
Recommendation: Add: "Each getSession() call returns a unique, single-use token. Generate a new session per transaction."

🏁 Overall rating

7/10

Good — targeted improvements would make it best-in-class for LLM-assisted integration

SEON's docs are well above average for a fraud-as-a-service platform. The Quick Start selector, modular API design, and clear scoring thresholds stand out. The primary gap — for both human developers and LLMs — is the lack of a single canonical end-to-end flow resource. Addressing the top 3 recommendations would make SEON one of the easiest fraud APIs to integrate via AI-assisted tooling.
